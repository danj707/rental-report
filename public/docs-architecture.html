Architecture & Admin Guide

Technical reference for the rec.us Enhanced Reports platform — how data flows, how reports are wired, and how to onboard new organizations.

System Overview

rec.us Reports is a single-file Node.js/Express application (server.js, ~9,200 lines) deployed on Railway Pro. It transforms raw Metabase/PostgreSQL data into interactive dashboards with AI-powered insights, PDF export, email subscriptions, and cross-org analytics.

Request Flow

Admin clicks report link → Token gate middleware → Static HTML (React/Babel CDN) → Client fetches /api/data → Server proxies Metabase → JSON response rendered

Stack

Shared UUID Architecture

Most reports use a single shared Metabase question with an {{org_id}} template variable. The server injects the org's UUID at query time. This means new orgs get all shared reports automatically — no new Metabase cards needed.

// SHARED_UUIDS — one Metabase card per report type, filtered by org_id
const SHARED_UUIDS = {
  facility: "f6787f45-...",
  programs: "e35f2b47-...",
  gl:       "4374b344-...",
  // ... 15+ report types
};

// Per-org overrides (e.g. Norman has a custom GL query)
norman: {
  gl: { mbUuid: "46b7e83b-..." },  // uses this instead of shared
}
Key rule: All report availability checks must use (org[r]?.mbUuid || SHARED_UUIDS[r]) — never just one or the other. The main data route prefers shared UUIDs + org_id for non-GL reports.

Onboarding a New Organization

Adding a new org requires only 4 fields in the ORGS map. All shared reports light up automatically via SHARED_UUIDS.

Minimum Config

neworg: {
  token:   "randomToken16chars",   // access token for URL auth
  orgId:   "uuid-from-rec-admin",  // rec.us organization UUID
  logoUrl: "https://...",          // org logo for report headers
}

Onboarding Steps

1. Get org UUID from rec.us admin → 2. Generate access token → 3. Add to ORGS map in server.js → 4. Push to main (auto-deploys)

Per-org Metabase cards are only needed for reports that require custom SQL (e.g. historic buildings with org-specific location names). Everything else flows through shared UUIDs.

Post-Onboarding Checklist

API Route Patterns

GET  /:org/:report            → Serves public/*.html with ORG_CONFIG injection
GET  /:org/:report/api/data   → Proxies Metabase, returns { rows, meta }
GET  /:org/:report/api/pdf    → Puppeteer renders page → PDF download
POST /:org/:report/api/insights → AI analysis of report data
GET  /:org/api/pulse           → Daily KPI snapshot (24hr cache)
GET  /:org/api/calendar-analytics → Calendar funnel metrics
GET  /:org/metrics             → Per-org usage analytics page
GET  /                         → Admin dashboard (password-gated)
GET  /api/admin/feedback       → Cross-org feedback aggregation
