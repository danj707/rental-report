Security & Privacy Overview

How rec.us Reports protects partner data — access controls, data handling, PII safety, and infrastructure security.

Security Posture

✅ Read-Only Database Access ✅ No PII in URLs ✅ Per-Org Token Isolation 🔒 Railway Pro Infrastructure 🔒 HTTPS Everywhere 🚧 Firebase OAuth (Planned)

The platform is designed around a zero-write, read-only architecture. Reports never modify the rec.us production database — all data flows one direction, from PostgreSQL through Metabase public card APIs to the reporting frontend.

Access Control

Token-Based Authentication

Every org has a unique 16-character access token. Requests to /:org/* routes must include the token as a query parameter. Invalid or missing tokens receive a generic 404 (not 403) to avoid leaking org existence.

Request arrives → Token middleware checks ORGS map → Match? Proceed → No match? 404

Org Isolation

Each org can only access its own data. Shared Metabase queries include an org_id parameter that filters results to the requesting org. There is no cross-org data access — even at the API level, an org's token only grants access to that org's routes.

Admin Dashboard

The admin dashboard at / is protected by a separate password. Sensitive operations (redeploy, report toggling, Metabase link editing) require re-entering the admin password.

Public Routes (No Token Required)

/:org/rentalcalendarPublic facility rental calendar — intentionally public for community access. PII stripped from all reservation data.
/:org/calendarPublic program calendar — shows class schedules only, no participant data.
/:org/api/trackAnalytics tracking endpoint for public calendars.

PII in Reports

Reports that display PII (names, emails, phone numbers) — such as Facility Rental Schedule, Class Roster, and Community Intel — are only accessible via authenticated org routes. The data is rendered client-side and never cached in a way that could leak across orgs.

PII Stripping for Public Routes

The rental calendar reservation overlay route explicitly strips all personal data before returning results. Only date, time, location, and booking status are included — no names, emails, or contact info.

AI Safety: When sending data to the Anthropic API for insights, the system sends only aggregated numbers (revenue totals, enrollment counts, category breakdowns). Individual-level PII (names, emails, addresses) is never included in AI prompts.

Audit & Observability

Event Logging

Every significant action is logged to /data/events.jsonl with timestamp, org, report type, event type, and metadata. Events include: page views, data fetches, PDF exports, Excel downloads, AI insight requests, email deliveries, and feedback submissions.

AI Observability

All AI API calls are traced via OpenTelemetry instrumentation to Langfuse (us.cloud.langfuse.com). Every insight, QBR narrative, and chat response has a trace ID linking to the full prompt, response, token usage, latency, and cost. Thumbs up/down feedback is attached to traces as Langfuse scores.

Health Checks

Automated health checks run on a tiered schedule (hourly for critical reports, every 6h for standard, daily for low-priority). Each check verifies Metabase card accessibility and response shape. Results are displayed on org cards in the admin dashboard with green/yellow/red status indicators.
